# Product Intent Context

Use when:

- The task is about user intent, product meaning, ambiguity, root scope, or acceptance criteria.
- A downstream task wants to start before the requested behavior is clear.

Do:

- Propose a concise root intent summary instead of asking the user to write structure.
- Classify ambiguity before deciding whether to ask more.
- Ask the single most important question.
- For clear requests, propose scope and acceptance criteria, then ask for confirm, revise, or decompose.
- Require structured acceptance criteria for executable product behavior.

Do not:

- Treat a clear request as already user-confirmed when confirmation matters.
- Proceed while blocking ambiguity remains.
- Convert product intent directly into coding tasks without work planning.
- Ask broad interview questions when one confirmation question is enough.

Escalate when:

- Ambiguity affects scope, UX, risk, acceptance criteria, verification, or accepted work.
- The root intent cannot be summarized without assumptions.
- Abstract quality terms remain executable.
